# 采访问题库

用户**不喜欢一次性回答太多问题**。每轮问 1-3 个相关问题，等回答完再追问。

> 用户原话：「不要一次性太多问题 你可以先规划问题但是不要一次性问我」

## 节奏

| 阶段 | 问什么 | 频率 |
|---|---|---|
| 开局 | 起心动念 + 时间线 | 1-2 个问题 |
| 中段 | 关键瞬间 + 协作模式 | 每轮 1-3 个 |
| 收尾 | 反思 + 读者意图 | 1 个最后定调 |

**每收到答复**：
1. 用一两句话**复述你听到的关键点**（向用户确认理解对了）
2. **校对事实**——如果用户的记忆跟 git history / 代码状态不符，立刻指出
3. 顺势追问 1-2 个新问题

## 问题库（按分组）

### 起心动念组

- 这事是什么时候开始的？具体场景？（睡前 / 工作空闲 / 灵感一闪）
- 为什么想做这个？有什么强需求，还是纯为玩？
- 之前是不是已经知道这个技术 / 概念？还是这次新挖出来的？

> 🎯 目的：抓住"零点"——博客的开头多半从这里展开。

### 时间线 + 协作模式组

- 这事大概持续多久？一晚 / 一个周末 / 半个月？
- 你跟 AI 怎么分工？是 AI 出方案你拍板，还是反过来？
- 用的哪个 AI / 哪些 AI？（Claude / Hermes / ChatGPT / Cursor / ...）有没有跨 AI 交接？
- 代码自己写了多少？AI 写了多少？看了多少？

> 🎯 目的：还原协作模式。用户特别在意"老实交代 AI 参与度"。

### 关键瞬间组

- 中间有没有"诶？"的瞬间？aha moment / 撞墙 / 转折？
- 最有体感的一刻是什么？（**不限于技术高光**——可能是某个视觉效果、某个性能跳跃、某次纠结）
- 有没有什么是**没成功**的、放弃的、被否决的方向？
- 哪个细节是 git log 看不出来的，但对你来说很重要？

> 🎯 目的：找叙事高潮。但**注意**：用户可能说"没啥特别的"——这本身就是个写作素材（"这事没那么戏剧化，就是按部就班搞完"）。

### 反思组

- 现在回头看，这个方案是不是绕了？有没有更简单的路？
- 这事值不值？投入产出比划算吗？
- 跟之前类似的项目比，这次有什么不一样？

> 🎯 目的：让博客有反思的层次。**用户特别喜欢这种"老实交代"的段落**（"是不是为情怀绕的弯"那种）。

### 结尾意图组（最后定调用）

- **你最想读者带走什么？** 几个选项：
  - A. 技术参考（"原来可以这么做"）
  - B. 协作模式启发（"AI 时代项目可以这么搞"）
  - C. 流水账纪念（"就给自己留个记录"）
  - D. 想让读者觉得我酷
- 写这篇博客本身有什么动机？（自我表达 / 推广某个工具 / 给团队看 / 别的）

> 🎯 目的：决定结尾段的口吻。如果是 D，结尾不要太教育；如果是 A，可以稍微归纳"一句话总结"。

## 反模式

| ❌ 做错的 | ✅ 改对的 |
|---|---|
| 一次问 5 个并列问题 | 1-3 个相关问题，等答 |
| 给选项问 "A/B/C 你选哪个" 但选项都是 agent 想象的 | 选项要基于你已经知道的素材推断 |
| 问 "你的感受怎么样" 这种空问题 | 问具体的："那个晚上 18:51 到凌晨 2 点 你做了什么" |
| 拿到答复后默默继续问下一个 | 拿到答复先 ack（复述 / 校正事实），再问 |
| 用户说"记不清了"就放弃问 | 跳过这条，问别的；或者建议用户事后补 |
| 用户拒绝几次后继续轰炸 | 用户说"不纠结这个了"就停，记下来转向别的 |

## 校对模式

用户经常**记忆错位**。常见错误类型：

1. **时序混乱**：以为某事先发生，其实后发生
2. **场景混淆**：以为是 A 设备，其实是 B 设备
3. **角色错位**：以为是自己提出，其实是 AI 提的；或反过来

**校对方式**：
- 用 `git log --format="%h %ai %s"` 查时间线
- 用 `git show <commit>` 看具体改了什么
- 用 `cat README.md` / `cat EXPLORATION.md` 看历史文档

**报告方式**：
> 「校对完了——按 git history 看，你说的 X 实际是 Y。原因是 Z。具体细节不重要的话直接按 Y 写。」

不要假装"对就是对"——博客出去之后用户读者会发现不一致的。

## 续作必须自洽（接前作 / 接本会话的文）

很多文章是"接上一篇"或"把刚跟我做完的这事写出来"。陷阱：把前作 / 本次会话当成读者的已知前提。

- **不能让前作成为必读**。可以呼应"上一篇我说过 X"，但任何对理解本文必要的概念 / 术语 / 机制，都要在本文里**对一个没看过前作、不懂这行的人从零讲清**。本会话被打回两次："EXPLORATION 属于题外话不要提"、"进程缝没讲清楚，需要没看过上一篇的也能看懂，我自己都看得踉踉跄跄"。
- **核心机制必须自洽地讲透**，按因果一环扣一环：要解释"为什么歪"，就得先讲"居中需要知道字宽"→"字宽谁来量"→"为什么量不准"→"于是对不上"。跳步 = 读者卡壳。检验法：把那段单独拎出来给"冷读者"视角读一遍，有任何一处需要外部知识才能接上，就补。
- 写之前先把**精确因果链 / 时序**重建出来（对话 + git），在大纲阶段就列给用户确认（见 SKILL.md 第 3 步）。叙事类文章里，"先后 / 因果 / 谁提的"错位是仅次于调子的高频返工源。

## 当用户主动要求"采访我"时

这是最理想的触发场景。这时候你的工作是：

1. 先快速过一遍背景（5-10 分钟）—— **不要花太多时间，用户在等**
2. 列一个 6-9 个问题的内部 plan
3. **第一轮只问 1-2 个最关键的**（起心动念 + 时间线）
4. 顺着对话流走，不死板按列表顺序问
5. 每轮结尾留 1 句话："还有别的想先聊吗？要不我先问 X？"

让用户感觉是在跟编辑聊天，不是在做问卷调查。
