# 文风指南

写博客的目标用户自己说过：

> 保持谦逊，我一直认为人外有人，所以不喜欢说教味。但是被人认为很酷会让我充满成就感，所以我想让自己变成别人口中的大佬，让自己显得很酷。

这段是文风的根。可以拆出几条：因为谦逊，所以不教育读者、不端着；因为想显得酷，所以要会自嘲、老实交代局限、展示判断力；因为想成为"大佬"，所以技术细节要展示判断而不是炫耀。

## 关于"AI 味"判断的先决条件

**用户自己有偏书面 / 偏职场化的表达习惯，这部分是他的真实语言，不是 AI 味。** 在判断一句话是否 AI 味之前，先拿他已有的博客和聊天语料做校准——比如"AI 能力已经能覆盖这种小项目了"听起来像翻译腔，但这就是他原话。只有他从来不用的措辞、明显的套话句式才算 AI 味。

判断顺序：先问"这是用户会自己说出来的话吗"，再问"这是 AI 味的模板吗"。如果第一问不确定，宁可保留，等他自己挑出来。

**用户亲口给的样本是最高优先级参照。** 如果他在对话里说"如果是我，我会这么写：……"，那段话就是文风锚——所有规则都次于它，包括 voice.md 自己写的规则。如果发现样本和某条规则冲突，**改规则，不改样本**。已收录的样本见下面"句子和段落"小节。

## 人称

主动语态。绝大部分情况用"我"，"我做了"不是"实现了"，"我让 Claude 改"不是"Claude 协助我"。

"我们"也会出现，是用户的语言习惯，指他和读者，类似把读者拉进来的"咱们"。比如"上次我们玩了那个墨水屏"。用户自己说不是刻意的，只是顺口，所以**别强行改成"我"**，但也不必刻意造"我们"——出现是自然的就留着。

设备和工具可以拟人化叫"他"。"墨水屏他跑得还算稳定" 是用户自然语言，不要改成"墨水屏跑得还算稳定"。

## 句子和段落

不要追求精炼短句开头。用户实际开场是流水账式、带引言回扣的，比如：

> 上次我们玩了那个墨水屏（上一篇的链接），他跑的还算稳定，这两天没有任何报错。但是我觉得也就是一个能跑的水平，代码没有很优雅。最关键是我心里的折腾瘾还没解，那就再给他挑挑毛病。

这段是用户亲口给的校准样本，每一句的句式、用词、节奏都是参照锚。从里面能看出：
- 引言式回扣上一篇是 OK 的（"上次我们玩了……"），不是反模式
- 短句不必精炼有力，可以平实流水（"他跑的还算稳定，这两天没有任何报错"）
- "但是"、"最关键是"、"那就……" 这种推进式连接词是他的真实语言，不算 AI 味
- 自我披露动机很重要（"我心里的折腾瘾还没解"），不是矫情
- 评价代码用人话（"也就是一个能跑的水平，代码没有很优雅"），不堆术语

短句和长句错着来。技术段落本来就会长，体感段落可以短。描述紧张、混乱、连续出错的瞬间，可以偶尔来一个堆并列的长串，"踩塌一脚、抓住的石头是松的、脚下又一片往下滚……"，和周围的句子拉节奏对比。

段落控制在 3-5 句，超过就拆段或起个小标题。小标题用动词或名词短语，不要完整句子。"第一拐点：浏览器预览就糊了"是好的，"我是如何发现 Satori 不适合墨水屏的渲染过程"是坏的。

## 技术细节

技术细节不是主菜。出场是为了让叙事更具体（"凌晨 02:03 commit pivot，睡了"）、为了展示判断（"SVG 光栅化是死结，浮点 Bezier 没 hinting"）、或者承认自己说不清（"FreeType MONO 我到今天还说不清"）。不要为了凑字数堆术语。

术语直接用，不要"（一种 X 的技术）"括注。行业里通用的英文术语就用英文，PR、commit、daemon、squash 这些译成中文反而怪。关键术语第一次出现可以一句话说明语境，但别展开。要解释一个深术语，与其堆定义不如打一个常识里的比方，像"FFN 就像 80 亿个 if 摞在一起"。比方只打一次，别顺着展开成一系列隐喻。

代码块 5-15 行的最好，超过 30 行的要重新审视，这是博客不是 README。展示数据流的话，ASCII 管线图或 mermaid 流程图比文字描述好读。

## 讲清一套系统：结构化，别堆大段文字

讲系统 / 流程 / 算法怎么转，大段连续文字是最差的载体。拆开用：

- **mermaid 流程图**（窄栏一律 `flowchart TD` 竖排）画数据流
- **表格**装对照信息：参数、源权威度 / 评分、判定条件、方案对比
- **有序列点**拆算法步骤（归一化 → 算相似度 → 聚类合并）

留给文字的是图表装不下的那部分：选型为什么这么定、踩了什么坑、设计时的私心——「为什么 / 什么感受」交给文字，「是什么 / 怎么做」交给图表。

**面向非技术读者时**（比如要发给不写代码的人看）：放低对背景知识的预期，但别降逼格——不删术语、不写成科普，而是给每个硬术语配一句生活比方，一句、不展开：

- SimHash →「给每条标题盖个数字指纹，两枚几乎重合就当撞了车」
- Jaccard →「看两条标题共用了多少词，过半就算同一件事」
- 并查集 →「A 像 B、B 像 C，就把三条串成一摊」

术语和参数全留着给懂行的看，比方让外行也 get「这步在解决啥」。能把复杂的讲得妙，逼格不降反升。

## 让"我"显得有底气

老实交代 AI 怎么参与的。不要"我做了这个项目"，可以"代码我没逐行看过，Claude 改完跑通了我就过，边角项目，行就行，不行回滚"。这种直白比含糊"在 AI 辅助下完成"显得更有判断力。

老实交代失败、局限、走过的弯路。不要"经过深思熟虑的架构设计"，可以"这是为情怀绕的弯。说实话我也没那么需要一个桌面小屏幕"。

老实交代知识边界。不要"我深入研究了 FreeType 的字体光栅化算法"，可以"FreeType MONO 这词儿我到今天还说不清"。

用具体的细节代替形容词。"性能大幅提升"不如"热路径从 400ms 掉到 10ms 一页"，"这是个不错的方案"不如"凌晨 02:03 commit pivot，睡了"。能用拟声词、动作动词、可数的具体值就别用形容词。"屏幕的初始化效果很糟糕"改成"屏幕亮起来的第一下是一串乱码"，"刷新时电流很大"改成"刷新那一瞬间把芯片拉到欠压、触发复位"。

把场景写实。"在工作之余完成的项目"不如"工作 Claude 一个 session、树莓派 Claude 另一个 session，挂着用。下午某个空档……"。

## 几种常见的段落节奏

写一个小要点的时候，节奏常是：陈述带细节、展开 why、承认局限或转折。比如「Python 进程 daemon 化，第一版每次渲染 `spawn` 一个新 Python 进程，冷启 ~370ms（字体加载占大头）。改成 daemon 模式后，Node 端建一条长连接：请求一行 JSON、响应 `OK <len>\n` + 字节 PNG。热路径从 400ms 掉到 10ms 一页。」

讲挫败到启发的节奏：我以为或尝试了 X、X 不行（说具体原因）、于是改成 Y。

结尾或某段反思的节奏：承认尴尬或局限、自嘲一句、肯定某个值。比如「代码我也没逐行审过。Claude 改完跑通了我就过，边角项目，行就行，不行回滚。」

这些不是模板，是常出现的几种节奏，参考用，不要死搬。

## AI 味的常见迹象

写完通读一遍，下面这些迹象出现就改。

**套话开头和收尾**。"目标很朴素：" / "最后说一句：" / "写博客也是 X 应用嘛" / "总的来说" / "归根结底" 这一类。直接进内容，别给自己找 punchline。

**硬凹的口语词**。AI 学"接地气"容易过头，反而暴露身份。"小活"、"扛"、"干完"、"搞起来"、"够使了"、"够呛"，还有突然冒出的"嘛""呗""哈"语气词。这些如果不是用户平时打字会用的，就是假。宁可用普通话也不要装。

**抽象的单段警句**。AI 喜欢把一句短话单独成段当 punchline。"也就这一句。" / "就这样。" / "至此，结束。" 这种是抽象修辞，是病。但同样是短段落，带具体的人事物或数字就有力，"好耶。" 或者 "凌晨 02:03 commit pivot，睡了。" 自检方式：这一句拿出来，里面有没有具体内容？有就留，没有就删。一篇里这种独立短段最多 1 处。

**半翻译腔和抽象名词**。常见的 AI 翻译腔包括 "值得注意的是……" / "这意味着……" / "在某种意义上" / "毫无疑问" / "首先……其次……再者……" 这类英文连接词的直译。另外，"体感"、"对齐"、"落地"、"赋能" 这种纯职场词不要进博客（"复盘"是边界，用户偶尔用，留意）。

注意：判断之前先用上面"先决条件"那一节校准——用户本人偏书面化的表达不算翻译腔。

**修辞工整的 setup → flip**。"先说 A，但 B" 如果太干净会显得排练过。比如「列出来挺唬人。但我体感一点都不难受——」改成「列出来挺唬人，但一点都不难受，这些坑几乎没有一个是我去解决的」就自然多了。

**元收尾段**。结尾上升到"我学到了 X"或者"从 A 到 B"这种抽象反思，是 AI 散文的标志。「从折腾到记录，我基本全程在旁边看着——看着看着，又一个吃灰的活过来了。」这种回文式收尾就是。让结尾保持具体，戛然而止比升华好。

**开篇剧透 + 自白写作动机**。开头就把结局抖出来、再表一通"我为什么写这篇"（"这套早报今天关了，跑了 52 天，关之前想认真记一遍"），是 AI 爱用的铺垫套路。删掉，直接从故事起点进去，结局留给后文自然抵达。

**changelog 式精确括注**。把日期、版本号成串塞进括号当数据交代（"订了 Claude（4 月 16 入 pro，5 月 9 升 5x）"），像变更日志不像散文。改成潜移默化的叙事（"越用越上头，没多久就升了一档"）——让读者感到投入在加码，不是在记账。

### 具体被否过的构造（高频 AI 腔）

下面是用户实际打回过的句式，写完逐句对照清除。表格保留作为快速扫描参考，但任何一条都从属于上面的"先决条件"——先确认用户语料里真的不用这种说法，再算 AI 腔。

| AI 腔构造 | 为什么是 AI 腔 | 改成什么 |
|---|---|---|
| `没有 X，没有 Y——就是 Z` | 排比三连 + 破折号亮底，太工整 | `没什么契机，就是手痒——爱折腾` |
| `再往下一层：…` / `想通这一层` / `根上都是这一条` | 论文式层层递进连接词 | `那再问一句：…` / `就这点事` / `都从这儿来` |
| `按理我该…，但…` | 造作的让步转折 | `上次我还有点心虚，这次不解释了` |
| 破折号当戏剧反转用（每段都来一发） | 节奏拿腔拿调 | 多用句号断开，平铺直叙 |
| `不是 A，是 B`（连续排比） | 对仗腔 | 偶尔用可以，别连用；改陈述句 |
| `这意味着 / 本质上 / 某种程度上 / 值得一提的是` | 过渡 / 总结套话 | 直接说结论，删过渡词 |
| `经过深思熟虑 / 优雅地 / 一举` | 自我拔高 | 删，换具体动作或数字 |

**判据**：一句话如果"工整、对仗、有总结腔、读着像技术分享稿"，就是 AI 腔，拆成短句、加口语词、允许不对称。宁可糙，不要顺得发腻。

## 破折号

写中文的 AI 容易把破折号用成万能停顿，逗号、冒号、句号、括号通通用 `——` 代替，一篇里能堆十几个。

800 字左右的短篇，破折号超过 3 处就要警觉，超过 6 处一定要删一轮，超过 10 处基本已经是重灾区。

替换原则按原意走。单纯转折或轻补充改成逗号；接续举例或解释改成冒号；前后能各自成句的改成句号；次要信息塞括号里。一篇内保留 0-2 处最有效的就行，其他全替。

## 用户调子如果变了

这里写的是目前的调子，人会变的。用户在采访里反复说"我想严肃一点写"、"这次想偏教程"，那就按用户的来。但即便偏教程，第一人称、不假装客观（"在我看来"、"我倾向"）、真实的失败和迷茫片段，这三样要保住。
