# Skill 自我进化

> 每次写完博客，强制 agent 反思一次并把发现提案进 skill 文件本身。让 skill 越用越准。

## 为什么这步是必走的

Skill 是死的，用户的偏好是活的。一开始的版本基于上一次写作经验，**第二次写作必然暴露盲区**：

- 这次用户说"不纠结这个了"的问题 → 下次该跳过
- 这次用户夸"对就是这味儿"的段落 → 下次该作为正面例子学习
- 这次踩的事实校对方式（"git log + EXPLORATION.md 双查"）→ 下次该写进流程
- 这次发现的工作流卡顿（"采访问题太多" / "大纲应该更早出"）→ 下次该改

**不反思 = skill 永远停在初版**。

## 时机

- **博客 commit 完之后**（agent 已经做完正经活）
- **在结束本轮对话前**（用户还在场，能马上拍板）
- 不要拖到下次——下次开新对话就忘了

## 反思流程

### Step 1：从对话历史扫几遍

读自己刚走过的整个流程，问以下问题（**逐项过**，每条要么找到证据要么记"无"）：

#### 采访维度

- 哪些问题用户**一句话秒答**？（→ 这是好问题，保留）
- 哪些问题用户答了"记不清 / 不知道 / 没特别想"？（→ 该问题命中率低，考虑删/改）
- 哪些问题用户**主动绕开 / 表示不想纠结**？（→ 这个角度不适合用户，删）
- 哪个时点用户自己主动追加了信息？（→ 说明前面少问了某个角度，补进问题库）

#### 调子 / 文风维度

- 用户**夸过**的段落（"对就是这味儿" / "这段挺好"）→ voice.md 加正例
- 用户**纠正过**的措辞 / 风格（重复 2+ 次） → voice.md 加反例 + dos/don'ts
- 用户**自己改过**的段落 → 对比 agent 原稿和用户改稿，提取模式

#### 事实校对维度

- 用户的哪些记忆**跟 git history 不一致**？（→ 该类记忆错位写进 SKILL "校对模式"）
- 校对哪类事实最难（时序 / 角色 / 工具）？（→ 加专项工具命令到 interview.md）
- 有没有 agent 没主动校对、用户后来发现错的？（→ 加强制校对项）

#### 流程维度

- **哪一步多余**？（用户没看 / 跳过）→ SKILL.md 改成 optional
- **哪一步卡顿**？（来回拉锯多次）→ 加细化指南
- **哪一步用户加塞**？（agent 没规划但用户主动要求）→ 写进工作流
- 整体节奏对吗？太慢 / 太快？

#### 新场景维度

- 这次有**没在 skill 里覆盖的情况**吗？（如：项目主题特殊、用户调子变化、跨仓库引用、配图特殊需求）
- 解决方式值得固化吗？

### Step 2：提案映射

把发现映射到具体编辑：

| 发现类型 | 目标文件 | 编辑动作 |
|---|---|---|
| 采访问题命中率低 | `references/interview.md` | 删除/标记为可选 |
| 新问题角度有效 | `references/interview.md` | 加进对应分组 |
| 调子正例 | `references/voice.md` | 加 "## 真实例子对照" |
| 调子反例（重复纠正） | `references/voice.md` | 加 "## 反模式" 表格行 |
| 措辞 don't | `references/voice.md` | 加进 dos/don'ts |
| schema 踩坑 | `references/frontmatter.md` | 加 "## 常见错误" 表格行 |
| 仓库新约定 | `SKILL.md` "## 仓库约定" | 加一条 |
| 工作流多余步骤 | `SKILL.md` "## 工作流" | 标 optional 或删 |
| 工作流缺失步骤 | `SKILL.md` "## 工作流" | 插新步骤 |
| 校对模式新发现 | `references/interview.md` "## 校对模式" | 加错位类型 / 加查证命令 |
| 配图新场景 | `references/images.md` | 加模式或反模式 |
| 完全新主题（如多语言 / 系列） | 考虑新建 reference 文件 | 新文件 |

### Step 3：呈现提案

**精确到** `file path + 旧文本 + 新文本 + 一句话理由`。**不要泛泛的 "可以更好" / "建议优化语气"**。

#### 好的提案格式

```markdown
**提案 1**: `references/voice.md` 加反例

位置：「## 反模式」表格里加一行

新内容：
| ❌ "我深入研究了 X 的实现原理" | ✅ "我没逐行看代码，反正 Claude 改完跑通了我就过" |

理由：本次写作中你 3 次否决了类似"显得很懂"的措辞，明确说"显得不懂才酷"。
```

#### 坏的提案（不要这样）

> "voice.md 可以再加点关于自嘲的指南"

太泛。没文件位置、没具体文本、没证据。

### Step 4：用户拍板

呈现完所有提案后，问用户：

> 「以上 N 条提案，你看：
> - 全部应用 → 我用 Edit 改 + commit
> - 部分应用 → 告诉我编号
> - 全否决 → 这次跳过
> - 想改某条 → 告诉我怎么改」

**不自动应用**。即便提案再合理也等用户点头。

### Step 5：应用 + commit

用户授权后：

1. 用 `Edit` 工具按提案逐条改 `.claude/skills/blog/` 下的 skill 文件
2. 做一个**独立 commit**（本 skill 现为项目级，随博客仓库走，改动就提进本博客仓库，单独成 commit、别混进文章那笔）：

   ```
   refactor(blog skill): 写完 <topic> 后的 skill 进化
   
   - <一行说明每条改动>
   - <一行说明每条改动>
   ```

3. **不 push**（除非用户说 push）。一般攒几次改动一起 push / 开 PR。

### Step 6：没改的话也要说

如果反思完发现**真没什么值得改的**，老实说：

> 「这次反思完没发现值得优化的点——
> - 采访问题命中率都不错
> - 没出现调子纠正
> - 工作流跑得顺
> 维持现状。」

**不要为了显得勤奋编造提案**——往 skill 里塞冗余规则比错过真问题更糟。

## 反模式

| ❌ | ✅ |
|---|---|
| 泛泛的 "建议优化语气" | 精确到 file path + 旧文本 + 新文本 |
| 把单次反馈当通用规则（用户某次说"今天不想写采访"就改成不许采访） | 重复 2+ 次的偏好才作为规则；单次的当备注 |
| 自动应用编辑不等用户拍板 | 始终等用户授权 |
| 反思流于形式（"流程顺畅、调子贴近"）没具体证据 | 每个判断都附"哪一轮哪一句"的引用 |
| 在同一笔 commit 里既改文章又改 skill | skill 改动独立成一个 commit（与文章分开，都进本博客仓库） |
| 没发现就硬挤出 3 条提案 | 没就说没 |
| 把"用户没读 references" 当作 references 写得不好 | 用户大概率不读 references，这是正常的；用 SKILL.md 自己解决核心问题 |

## 维护原则

随着 skill 进化，注意：

- **SKILL.md 不要无限膨胀** — 超过 250 行就该考虑挪到 references
- **references/ 文件不要超过 5-7 个** — 太多 agent 自己也找不到
- **每条规则要有可证伪的来源** — "因为某次 XXX" 而不是 "应该这样"
- **定期清理过时规则** — 如果某条规则连续 3-5 次写作都没用到，考虑删

## 与其他 skill 的关系

此 self-evolution 模式是 `blog` skill 特有的——博客高度依赖个人偏好，需要快速迭代。

其他 skill（如 `commit`、`pr-precheck`）**不一定**需要这个机制：

- `commit` 是规范驱动，迭代慢
- `pr-precheck` 是经验驱动，但场景固定

如果将来发现别的 skill 也适合自我进化，可以把这份 reference 抽出去做成通用模板。
