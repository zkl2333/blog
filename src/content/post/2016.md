"某天"不够准确。就是昨天晚上 10 点半开始的。

于是顺着 `run_agent.py` 一路追下去。下面是我从**触发条件 → Fork 子 Agent → Review Prompt 设计 → 调用计数 → 遗忘机制 → Token 成本**的全链路追踪记录。所有行号基于 `hermes-agent` 当前 `main` 分支。

## 一、完整流程图

先上全景，后面逐段拆解：

```
┌─────────────────────────────────────────────────┐
│            run_conversation() 入口               │
│                                                  │
│  [TURN 开头] Memory Nudge 检查                   │
│    _turns_since_memory += 1  (每个用户消息)       │
│    if >= memory.nudge_interval → 标记待 Review    │
│                                                  │
│  [主循环] 模型 API 调用 + 工具执行...             │
│    每轮 tool iteration:                          │
│      _iters_since_skill += 1                     │
│                                                  │
│  [TURN 结尾] Skill Nudge 检查                    │
│    if _iters_since_skill >= skills.              │
│       creation_nudge_interval → 标记待 Review     │
│                                                  │
│  ┌── 如果任一标记为 True ─────────────────────┐  │
│  │                                            │  │
│  │  _spawn_background_review()                 │  │
│  │    │                                       │  │
│  │    ├─ 选 Prompt (MEMORY/SKILL/COMBINED)    │  │
│  │    ├─ threading.Thread → 子线程            │  │
│  │    │   ├─ approval_callback = auto-deny    │  │
│  │    │   ├─ fork 新 AIAgent:                │  │
│  │    │   │   model/credential = 继承父 agent │  │
│  │    │   │   max_iterations = 8              │  │
│  │    │   │   toolsets = [memory, skills]     │  │
│  │    │   │   quiet_mode = True               │  │
│  │    │   │   自身 nudge_interval = 0 (防递归) │  │
│  │    │   │   _memory_store = 共享父的         │  │
│  │    │   │                                  │  │
│  │    │   └─ run_conversation(               │  │
│  │    │        prompt,                        │  │
│  │    │        conversation_history=完整对话   │  │
│  │    │      )                                │  │
│  │    │                                      │  │
│  │    └─ 扫描 result → 去重(按tool_call_id)  │  │
│  │       → 输出 "💾 Memory updated"          │  │
│  └────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

**两个计数器对比：**

| | Memory Nudge | Skill Nudge |
|---|---|---|
| **计数器变量** | `_turns_since_memory` | `_iters_since_skill` |
| **粒度** | 每个用户消息 +1 | 每轮 tool iteration +1 |
| **检查时机** | Turn 开始时（line 9744） | Turn 结束后（line 12986） |
| **配置路径** | `memory.nudge_interval` | `skills.creation_nudge_interval` |
| **默认值** | 10 | 10 |

几个关键规则：
- **跨 turn 不重置**：line 9697 明确注释 `NOT reset here`，同一 Agent 实例内持续累加
- **触发后归零**：Nudge 触发后对应计数器清零
- **Session 恢复时清零**：`_load_session_context()` 在 line 8746/9096 重置
- **子 Agent 的 Nudge = 0**：防递归，不然子 Agent 又 Fork 孙 Agent

## 二、可配置项一览

所有配置都写在 `~/.hermes/config.yaml`：

```yaml
# 控制 Memory Review 触发频率（多少轮对话触发一次）
# 源码：run_agent.py line 1617
# 默认值：10
memory:
  nudge_interval: 30        # 调低 = 更频繁检查 = 但增加后台 API 调用

# 控制 Skill Review 触发频率（多少次工具调用触发一次）
# 源码：run_agent.py line 1719
# 默认值：10
skills:
  creation_nudge_interval: 30

  # Skill Guard Agent：安装第三方 skill 前的安全检查
  # 源码：tools/skills_guard.py
  guard_agent_created: false
```

**这两个 interval 设多少合适？**

- 默认 10：每 10 次工具调用就触发一次后台 Review。意味着你让 Agent 干一件复杂事（比如调了 12 个 tool），结束后它大概率要 fork 一个子 Agent 去跑 Review，烧一轮 API 钱。
- 我设的 30：降低到每 30 次才触发，减少后台开销。适合日常使用。
- 设成 0：完全关闭后台 Review。但这意味着 Skill 和 Memory 都不会自动更新了。

**`_SKILL_REVIEW_PROMPT` 能不能自定义？**

不能。三个 Review Prompt（`MEMORY_REVIEW_PROMPT`、`SKILL_REVIEW_PROMPT`、`COMBINED_REVIEW_PROMPT`）全部**硬编码**在 `run_agent.py` 第 3147-3201 行，是类级别的字符串常量，没有暴露到 config.yaml，也没有环境变量或插件钩子可以覆盖。

这意味着你无法在不修改源码的情况下告诉 Review Agent"顺便检查一下哪些 Skill 三个月没用了，建议清理"。Prompt 让它干什么它就只干什么，而它目前只被要求创建和更新。

## 三、Skill Review Prompt 全文逻辑

`_SKILL_REVIEW_PROMPT`（line 3158-3181，我翻译并注释）：

```
1. SURVEY 现有 skill 格局
   先调用 skills_list 看全局，再 skill_view 可能有关系的
   你在找的是「任务类别」而非「这个具体任务」

2. THINK CLASS-FIRST
   刚刚完成了什么类型的任务？触发条件是什么？
   用一句话描述类别，再决定要不要存

3. PREFER GENERALIZING
   优先更新/扩展现有 Skill，而非新建
   如果已有 Skill 覆盖了这类任务（哪怕只是部分）→ patch 它

4. ONLY CREATE NEW
   只有当没有已有 Skill 合理覆盖时，才新建
   命名和范围都定在「类别级别」，而不是「这次会话」

5. 如果发现重叠 → 只标记，留给"未来的 Review"合并
   不要现在合并，除非重叠很显而易见且低风险

只有真正值得存的时候才行动。
没什么可存的话就说 "Nothing to save." 然后停。
```

注意第 5 条的措辞 —— _"Do not consolidate now unless the overlap is obvious and low-risk"_。合并不做、删除没有。整个 Prompt 的哲学是保守优先。

`_COMBINED_REVIEW_PROMPT`（Memory + Skill 双触发时用）的逻辑也一样，多了 Memory 部分，Skill 部分完全一致。

## 四、调用计数：被动式的"翻旧账"

Skill 使用统计不在运行时维护一个计数器，而是在 `agent/insights.py` 的 `_get_skill_usage()` 方法里：

```
打开 /insights 面板时
  → 查 session DB (SQLite)
    → 扫描历史消息中的 tool_calls
      → 过滤 tool_name == "skill_view" | "skill_manage"
        → 提取 args.name 作为 skill 名
          → 聚合: view_count, manage_count, last_used_at
```

这不是实时的。每次查询都现场扫一遍历史记录。好处是不需要额外持久化；坏处是慢，且只能当成事后审计用，无法用于实时触发清理策略。

## 五、所有 Skill 描述都拼进 System Prompt

另一个直接的成本问题：

```
build_skills_system_prompt()  ← prompt_builder.py line 650
  │
  ├── 扫描 ~/.hermes/skills/ 下所有 SKILL.md
  │     ├── 读 frontmatter: name, description, category
  │     ├── 过滤: 平台不匹配 → 跳
  │     ├── 过滤: 已 disable → 跳
  │     └── description 截断到 60 字符
  │
  ├── 两层缓存: in-process LRU + 磁盘 snapshot
  │
  └── 拼成 XML 块注入 system prompt:

  ## Skills (mandatory)
  <available_skills>
    autonomous-ai-agents: Skills for spawning...
      - claude-code: Delegate coding to Claude Code CLI...
      - codex: Delegate coding to OpenAI Codex CLI...
      - hermes-agent: Configure, extend, or contribute...
    devops: Skills for managing infrastructure...
      - cron-script-execution: Run Python scripts reliably...
      ...
  </available_skills>
```

关键：**你装的每一个 Skill，不管用没用过，每次 API 调用都进 system prompt。**

按 DeepSeek V4 Flash 当前定价（$0.28 / 1M output）算。数字本身不大，但有两点值得注意：

1. **只增不减** — 后台 Review 会自动创建新 Skill，从不删除旧的。这个成本是单调递增的。
2. **模型升级后更贵** — 现在用 Flash 感受不明显，但如果你用 Pro（$3.48/1M）跑，146 个 Skill 的 system prompt 部分一年就是 **$278**。

## 七、遗忘机制：搜遍源码的结论

| 机制 | 状态 | 位置 |
|------|------|------|
| 自动清理不活跃 Skill | ❌ | — |
| TTL / 过期 | ❌ | — |
| 按调用频率淘汰 | ❌ | — |
| disk-cleanup 插件 | ❌ 明确排除 `skills/` | plugins/disk-cleanup |
| 后台 Review 删除 | ❌ Prompt 没指令 | run_agent.py:3179 |
| 重叠合并 | ❌ 只标记不做 | run_agent.py:3176-3178 |
| 手动删除 | ✅ | `skill_manage(action='delete')` |

源码级验证：`_SKILL_REVIEW_PROMPT` 和 `_COMBINED_REVIEW_PROMPT` 里，**没有任何一个词涉及删除、清理、淘汰、归档。** Review Agent 被分到的工具集是 `["memory", "skills"]`，它手上有 `skill_manage(action='delete')` 的能力，但 Prompt 没让用。那它就不会用。

## 九、总结

```
触发：两个独立计数器
  │   memory.nudge_interval（按 turn）
  │   skills.creation_nudge_interval（按 tool iteration）
  │   两个配置项，可以调，可以关
  ↓
Fork：独立线程 + 继承凭证 + 工具白名单
  │   max_iterations=8, quiet_mode=True
  │   自身 nudge=0 防递归
  │   approval callback=auto-deny 防卡死
  ↓
Review：Prompt 指导 → 创建/更新（不删不合不归档）
  │   _SKILL_REVIEW_PROMPT 硬编码在源码里，不可自定义
  │   没有删除指令、没有过期逻辑、没有淘汰策略
  ↓
注入：每个 Skill 的 name + 60字符描述 → system prompt
  │   两层缓存，但内容照单全收
  │   不管用没用过，每次 API 调用都带
  ↓
膨胀：只增不减，无自动淘汰
  │   146 Skill ≈ 4,400 token/轮 ≈ $22/年（Flash）
  │   如果用 Pro 模型，成本翻 12 倍
  ↓
现状：5 个 Open Issue，0 个 PR
```

这不是 Bug，是设计取舍。保守哲学本身没问题——不主动删用户的东西是负责任的。但"自动创建"和"零淘汰"的组合拳，意味着系统会随着使用时间线性膨胀。

**生产环境建议：**

1. 定期 `hermes skills list`，手动清理从没用过的 Skill
2. 把 `skills.creation_nudge_interval` 调高（30-50），减少自动创建的频率
3. 把 `memory.nudge_interval` 同步调高，保持一致性
4. 关注 [#11425](https://github.com/NousResearch/hermes-agent/issues/11425) 的进展——如果它合并了，记得第一时间打开归档功能
