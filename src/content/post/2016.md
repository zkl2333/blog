某天我发现 Hermes 里多了几个我从来没要求过的 Skill。一番追溯后发现是后台自动创建的。于是从昨晚开始顺藤摸瓜，把整条链路从触发、Fork、计数到遗忘机制全追了一遍。记录如下。

---

## 1. 触发：两个计数器

入口在 `run_agent.py`，两个独立的计数器控制背景 Review 的触发时机：

### Memory Nudge

```
每个用户消息 → _turns_since_memory += 1
当 >= memory.nudge_interval (默认10) → 触发 MEMORY_REVIEW
```

检查在 `run_conversation()` 开头（line 9744），触发后计数器归零。

### Skill Nudge

```
每次 API 调用（tool iteration）→ _iters_since_skill += 1
当 >= skills.creation_nudge_interval (默认10) → 触发 SKILL_REVIEW
```

检查在 `run_conversation()` 结尾（line 12986），**等回复发给用户之后**才触发，不阻塞主流程。

两个计数器在同个 AIAgent 实例内持续累加，turn 之间**不重置**，只有触发后才归零。session 恢复时也会清零。

---

## 2. Fork：后台子 Agent

当触发条件满足，`_spawn_background_review()` （line 3267）拉起一个**独立的线程**：

```python
review_agent = AIAgent(
    model=父agent同模型,
    provider=父agent同provider,
    base_url/api_key/api_mode=完全继承,
    max_iterations=8,
    enabled_toolsets=["memory", "skills"],
    quiet_mode=True,  # stdout/stderr → /dev/null
)
review_agent._memory_nudge_interval = 0  # 防递归
review_agent._skill_nudge_interval = 0
review_agent._memory_store = 共享父agent的 # 直接写入
```

向它注入**完整对话历史** + Review Prompt，让子 agent 自己判断要不要存。

根据触发类型选择 Prompt：
- 双触发 → `COMBINED_REVIEW_PROMPT`
- 仅 Memory → `MEMORY_REVIEW_PROMPT`
- 仅 Skill → `SKILL_REVIEW_PROMPT`

安全措施：子 agent 的 approval callback 设为 `auto-deny`，防止执行危险命令时卡在交互弹框。

完成后扫描子 agent 的 tool 调用结果，按 `tool_call_id` 去重后输出 `💾 Memory updated` 或类似提示。

## 3. Skill Review Prompt 到底让它干什么

这是真正的"大脑"——`_SKILL_REVIEW_PROMPT`（line 3158）：

```
1. SURVEY 现有 skill（skills_list → skill_view）
2. THINK CLASS-FIRST（识别任务类，不是具体任务）
3. PREFER GENERALIZING（优先扩展现有 skill）
4. ONLY CREATE NEW（实在没覆盖才新建）
5. 如果发现重叠 → 只标记，不合并
```

注意最后一条：**发现重叠不合并，只记下来留给"未来的 review"**。连合并都没做。

更关键的是：**整个 Prompt 没有任何删除、淘汰、清理的指令。** 子 agent 手里有 `skill_manage` 工具，能力上是可以删的，但 Prompt 里根本没让它考虑这件事。

---

## 4. 调用计数：有，但是"查账"式的

Skill 使用统计不在运行时维护，而在 `agent/insights.py` 的 `_get_skill_usage()` 里：

```python
扫描 session DB 历史消息
  → 找 tool_calls 里 name == "skill_view" | "skill_manage"
    → 按 skill name 聚合:
        view_count、manage_count、last_used_at
```

这是**扫描历史记录**，不是实时计数器。每次打开 `/insights` 面板时现场算。

---

## 5. 遗忘机制：没有

搜遍源码，结论是：

| 机制 | 状态 |
|------|------|
| 自动清理不活跃 Skill | ❌ |
| TTL / 过期 | ❌ |
| 按调用频率淘汰 | ❌ |
| disk-cleanup 插件清理 | ❌ 明确排除 `skills/` |
| 后台 Review 删除 | ❌ Prompt 没指令 |
| 手动删除 | ✅ `skill_manage(action='delete')` |

Review Prompt 的第 5 步写的是"标记重叠留给未来合并"，本身就暴露了设计意图是保守的——宁可不合并也不误删。这逻辑延伸下来就是不删。

---

## 6. 所有 Skill 描述全都拼进 System Prompt

`prompt_builder.py` 的 `build_skills_system_prompt()` 扫描所有 Skill，把 name + description 拼成 XML 块塞进 system prompt。每个 description 被 `extract_skill_description()` 截断到 60 字符。

你安装的每个 Skill，哪怕从没用过，每次 API 调用都会被带进 system prompt。

---

## 7. Token 成本量化

社区里一位装了 146 个 Skill 的用户（#13534）算了笔账：

| Skills | Tokens/轮 | 年成本（50轮/天） |
|--------|-----------|-------------------|
| 58 个（我的） | ~1,500 | ~27M |
| 89 个 | ~2,300 | ~42M |
| 146 个 | ~4,400 | ~80M |
| 300 个 | ~9,000 | ~164M |
| 500 个 | ~15,000 | ~274M |

按 DeepSeek V4 Flash 价格（$0.28/1M output）算，146 个 Skill 的 system prompt 部分每年白白烧掉约 $22。不算多，但它**只增不减**——而且 model 越好、context 越长，这个成本就越贵。

## 9. 总结

```
触发：两个独立计数器（memory turns / skill iterations）
  ↓
Fork：独立线程 + 继承凭证 + 工具白名单
  ↓
Review：Prompt 指导 → 创建/更新（不删不合）
  ↓
注入：每个 Skill 的描述都进 system prompt
  ↓
膨胀：只增不减，无自动淘汰
```

这设计的问题不在于"后台自动创建"本身——那是个好特性。问题在于**没有与之配对的回收机制**。一个会自动生长但从不修剪的系统，最终会吃掉自己的 context 空间。

如果你也在用 Hermes，定期 `hermes skills list` 看看有没有垃圾 Skill 需要手动清理，别等它自己变聪明——目前它不会。
